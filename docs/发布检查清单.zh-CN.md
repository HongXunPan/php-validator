# 发布检查清单

本文档用于说明 `hongxunpan/validator` 在当前 **pre-1.0** 阶段的最小发布检查流程。

目标不是把发版流程做得很重，而是避免出现：

- README / CHANGELOG 没同步就发版
- public contract 变了但没有说明
- 本地能跑、CI 没绿就发版
- 把项目级兼容逻辑误当成 core 默认契约发出去

---

## 1. 适用范围

本清单适用于以下场景：

- 准备打一个新的 git tag
- 准备发布新的 Composer 版本
- 准备对外宣称“当前阶段可接入 / 可试用”
- 准备把某轮较大的公开行为变更作为一个正式节点对外说明

不适用于：

- 纯本地实验性提交
- 只在父工作区任务单里记录的未完成方案

---

## 2. 发版前先确认的边界

发布前先回答下面 3 个问题：

### 2.1 这次变更是否影响公开契约

如果影响以下任一项，就不只是内部小修：

- `Validator`
- `ValidationKernel`
- `ValidationResult`
- `RuleInterface` / Rule 扩展模型
- README 已文档化的 DSL / 用法
- `docs/公开契约与稳定性承诺.zh-CN.md` 中列出的公开边界

### 2.2 这次变更是否只是 internal 调整

如果只改了：

- `Internal/*`
- 测试
- CI
- 文档错字

且没有影响公开行为，可以按较轻的变更说明处理。

### 2.3 这次变更是否混入了项目级兼容逻辑

发布前必须再次确认：

- 是否把某个业务项目的 legacy helper 契约推回了 core
- 是否把 ORM / `unique / exists` 这类项目规则推回了 core
- 是否把项目 adapter layer 的样板误写进了公开包文档

如果有，先回退再发版。

---

## 3. 文档检查

发布前至少检查下面这些文件是否需要同步：

### 3.1 README

当出现以下任一情况时，README 必须同步：

- 新增公开入口
- 修改公开返回值语义
- 新增推荐用法
- onboarding 示例变化

要同时检查：

- `README.zh-CN.md`
- `README.md`

### 3.2 公开契约文档

如果 public / internal 边界变化，必须同步：

- `docs/公开契约与稳定性承诺.zh-CN.md`

### 3.3 CHANGELOG

只要这轮变更对外可见，就应至少补一条到：

- `CHANGELOG.md`

### 3.4 CONTRIBUTING

如果贡献方式、验证方式、提交要求变化，需同步：

- `CONTRIBUTING.md`

---

## 4. 代码与验证检查

### 4.1 最小测试

本地至少执行：

```bash
composer test
```

### 4.2 静态分析

如本轮涉及：

- 类型
- 继承关系
- 方法签名
- 目录迁移
- 公开扩展契约

则至少执行：

```bash
composer analyse
```

若当前执行环境（包括容器真相环境）尚未安装 dev 依赖，导致 `phpstan` 二进制不可用，则应满足以下二选一之一：

- 先补齐当前待发版环境的 dev 依赖，再执行 `composer analyse`
- 以当前待发版提交对应的 GitHub Actions `static-analysis` 绿灯作为最终放行依据

### 4.3 CI 状态

准备打 tag 前，至少确认：

- GitHub Actions 多版本测试已通过
- static-analysis job 已通过

如果 CI 未绿，不建议继续发版。

### 4.4 Composer 元数据检查

建议至少执行：

```bash
composer validate --strict --no-check-publish
```

目标是尽早发现：

- `composer.json` 基本结构错误
- script / metadata 声明异常
- 会影响发布可信度的基础配置问题

---

## 5. 仓库卫生检查

发版前确认：

- `.idea/`、`.vscode/`、缓存文件未误入 git
- `vendor/` 未进入仓库
- `.gitignore` / `.gitattributes` 仍符合当前发布边界
- 根目录没有临时调试文件

---

## 6. 版本与 tag 检查

### 6.1 当前阶段建议

由于项目仍处于 **pre-1.0**，当前约定：

- 首个 tag 只能是：
  - `0.0.1`
- 本轮不采用：
  - 更高版本号起步方式
  - `rc` 预发布命名

不要过早宣称 `1.0.0`。

### 6.2 tag 前确认

打 tag 前至少确认：

- 这轮变更已经有 changelog 记录
- README / docs 已同步
- 测试与分析通过
- public contract 是否变化已被说明

---

## 7. 对外说明建议

发布说明里建议至少写清：

- 这是 pre-1.0 阶段的哪个节点
- 这次主要新增 / 改了什么
- 是否有 public contract 变化
- 是否需要 adapter layer 项目特别关注迁移

如果某轮只是 internal 收口，不要夸成“大版本能力重构”。

---

## 8. 最小发布清单

每次正式发版前，至少过一遍下面清单：

- [ ] 本轮变更范围已确认：公开行为 / internal / 工程化 / 文档
- [ ] README 中英文已按需同步
- [ ] `docs/公开契约与稳定性承诺.zh-CN.md` 已按需同步
- [ ] `CHANGELOG.md` 已按需同步
- [ ] `CONTRIBUTING.md` 已按需同步
- [ ] `composer test` 通过
- [ ] 如适用，`composer analyse` 通过
- [ ] GitHub Actions 多版本测试通过
- [ ] GitHub Actions 静态分析通过
- [ ] 仓库无噪音文件、无临时产物
- [ ] 这次发布没有把项目级 legacy 兼容逻辑误推回 core
- [ ] tag / 版本号与本轮变更量级匹配
