# 性能与基准说明

本文档用于说明 `hongxunpan/validator` 当前阶段对性能、benchmark 与优化取舍的态度。

当前仓库仍处于 **pre-1.0** 阶段，现阶段的优先级不是“为了看起来快而过早微优化”，而是：

1. 先保证公开契约清楚；
2. 先保证执行链可理解、可测试、可维护；
3. 在出现明确热点后，再做可验证的性能优化。

---

## 1. 当前性能立场

当前 `hongxunpan/validator` 的默认立场是：

> **优先可维护性、可扩展性与规则语义清晰度；在不明显牺牲运行成本的前提下，再逐步做性能收口。**

因此，当前实现中有意识保留了：

- 结构化对象承接运行期 data 生命周期；
- 明确的阶段拆分：
  - materialize
  - conditional presence
  - presence
  - local value
  - dependent value
- 可读性优先的 internal 协作者分层；
- 明确的 public / internal 边界。

这些设计会比“所有东西挤在一个大循环里”更容易维护，但在极端大批量场景下，未必是绝对最省分配次数的形态。

---

## 2. 当前不追求的优化

在没有真实数据支撑之前，当前阶段**不追求**下面这些“看起来很快”的提前优化：

- 为了减少对象而回退到匿名数组黑盒传递；
- 为了少几层协作者而把 internal 重新压回大方法；
- 为了追求极限吞吐而牺牲 public contract 清晰度；
- 为了某个单项目的局部热点，把默认架构改成难维护的特化实现。

如果未来要做这类优化，必须先有 benchmark 或实际业务热点证据。

---

## 3. 什么时候应该补 benchmark

出现以下情况时，应优先补 benchmark，而不是继续凭感觉讨论快慢：

### 3.1 大量列表校验

例如：

- 单次请求里有很长的 object list
- 大量 item 都要走完整规则链
- 某些字段还有依赖型 compare 规则

### 3.2 多层嵌套对象

例如：

- 大量点路径读写
- `reject_unknown`
- 多层 `field_prefix`
- 列表包对象再嵌对象

### 3.3 某次 internal 重构明显改变分配模型

例如：

- 新增大量中间对象
- 新增额外遍历轮次
- 调整 `ValidationOutput` / `TargetValueContext` / `RuleContext` 承接方式

### 3.4 用户报告真实性能问题

例如：

- 校验耗时明显偏高
- 与历史 helper 相比回退明显
- 在低版本 PHP 下出现不可接受的吞吐下降

---

## 4. benchmark 的目标

未来补 benchmark 时，目标不是造一个很复杂的性能体系，而是至少回答下面几个问题：

1. 小对象校验场景是否足够轻；
2. 大列表场景下的增长曲线是否可接受；
3. 依赖型规则引入后是否明显放大成本；
4. 与同仓旧 helper 相比，成本变化量级大概是多少；
5. 某次优化是否真的有效，而不是只是换了一种写法。

---

## 5. 推荐的 benchmark 粒度

未来如果补 benchmark，建议至少覆盖 3 类场景：

### 5.1 小对象基础场景

- 3~5 个字段
- 无依赖规则
- 少量 normalize

### 5.2 中等对象场景

- 10~20 个字段
- 包含：
  - trim
  - blankToNull
  - minLength / maxLength
  - formatTime
  - compare 规则

### 5.3 列表场景

- scalar list
- object list
- object list + dependent rules

---

## 6. 如何看待 benchmark 结果

benchmark 结果不应只看“绝对谁更快”，还要同时看：

- 公开契约是否更稳定；
- 错误边界是否更清楚；
- internal 是否更可维护；
- 优化是否只对某个极端 case 有利，却破坏了大多数场景的可读性。

也就是说，当前项目的判断标准不是：

> “只要慢一点就一定要回退结构”

而是：

> “只有当性能成本达到不可接受，且优化方案可验证、可回滚、不会破坏边界时，才值得继续为性能改结构。”

---

## 7. 当前阶段的最小结论

当前阶段可以对外说明：

- 本包已重视运行期 data 生命周期与规则执行边界的清晰性；
- 当前实现优先保证维护性、可扩展性与可验证性；
- 已预留后续 benchmark 与性能优化空间；
- 在没有真实热点前，不以牺牲结构为代价做过早微优化。

如果未来引入正式 benchmark，应在：

- `CHANGELOG.md`
- `README`
- 或新增 benchmark 文档

中明确记录 benchmark 场景、命令与结果口径。
