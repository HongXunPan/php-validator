# 项目适配层接入样板

## 1. 文档定位

本文档给出一个**业务项目接入 `hongxunpan/validator` 的最小适配层样板**。

适用场景：

- 原项目历史上已经有自己的 validator helper / facade
- 现有调用方仍依赖数组 envelope
- 现有调用方仍依赖 `validateOrThrow()` 直接返回 payload 数组
- 项目内仍有 legacy rule / alias / 中文错误提示 / ORM 相关规则

本文档目标不是要求所有项目照抄同一份实现，而是提供一个**稳定的接法骨架**。

---

## 2. 接入原则

### 2.1 core 与适配层职责分开

`hongxunpan/validator` core 负责：

- 统一规则模型
- canonical 校验主链
- `ValidationResult` 对象语义

业务项目适配层负责：

- 旧 helper 契约接回
- 旧数组 envelope 接回
- `*OrThrow` 入口
- legacy rule / alias / 中文提示
- ORM / 业务相关规则，如 `unique / exists`

### 2.2 不追求无缝替换幻觉

如果历史项目原来拿的是：

- `['count' => ..., 'errors' => ..., 'detail' => ..., 'validated_data' => ...]`

或者原来直接写：

```php
$payload = Validator::validateAndNormalizeOrThrow(...);
```

那么应当在项目适配层中**显式接回**，不要要求 core 回退成旧心智。

### 2.3 不建议的兼容方式

不建议为了兼容而：

- 让 `ValidationResult` 实现 `ArrayAccess`
- 把 core 再做成数组优先 API
- 把某个项目的 helper 签名直接提升成公共包默认契约

---

## 3. 推荐目录骨架

一个典型项目可以按下面方式组织：

```text
app/
  Exceptions/
    ValidateException.php
  Helpers/
    Validate/
      Validator.php
      能力清单.md
  Validators/
    Rules/
      UniqueRule.php
      ExistsRule.php
      LegacyTimeAfterOrEqualFieldRule.php
```

其中：

- `Helpers/Validate/Validator.php`：项目适配层入口
- `Validators/Rules/*`：项目自己的业务规则 / legacy 兼容规则
- `ValidateException.php`：项目自己的异常翻译与抛错模型

## 4. 最小适配层样板

下面这份代码只示意**结构与职责**，不是要求项目逐字照抄。

```php
<?php

namespace App\Helpers\Validate;

use App\Exceptions\ValidateException;
use App\Validators\Rules\ExistsRule;
use App\Validators\Rules\UniqueRule;
use HongXunPan\Validator\Result\ValidationResult;

class Validator extends \HongXunPan\Validator\Validator
{
    protected static $extraRules = array(
        'unique' => UniqueRule::class,
        'exists' => ExistsRule::class,
    );

    protected static $ruleAliases = array(
        'egt' => 'gte',
        'elt' => 'lte',
        'lenMin' => 'minLength',
        'lenMax' => 'maxLength',
    );

    protected static $ruleMessages = array(
        'required' => '请输入$paramName',
        'string' => '$paramName类型错误',
        'minLength' => '$paramName长度不足$rule位',
        'maxLength' => '$paramName长度超过$rule位',
        'timeAfterOrEqualField' => '$paramName不能早于$rule',
        'unique' => '$paramName已存在',
        'exists' => '$paramName不存在',
        'unsupported' => '$paramName使用了不支持的校验规则：$rule',
    );

    public static function validate(array $data, array $rules, array $options = array())
    {
        return parent::validate($data, $rules, $options)->toArray();
    }

    public static function validateAndNormalize(array $data, array $rules, array $options = array())
    {
        return parent::validateAndNormalize($data, $rules, $options)->toArray();
    }

    public static function validateListAndNormalize(array $list, $rules, array $options = array())
    {
        return parent::validateListAndNormalize($list, $rules, $options)->toArray();
    }

    public static function validateOrThrow(array $data, array $rules, $throwFirst = true)
    {
        $result = parent::validate($data, $rules);

        return static::unwrapOrThrow($result, $throwFirst);
    }

    public static function validateAndNormalizeOrThrow(array $data, array $rules, array $options = array(), $throwFirst = true)
    {
        $result = parent::validateAndNormalize($data, $rules, $options);

        return static::unwrapOrThrow($result, $throwFirst);
    }

    public static function validateListAndNormalizeOrThrow(array $list, $rules, array $options = array(), $throwFirst = true)
    {
        $result = parent::validateListAndNormalize($list, $rules, $options);

        return static::unwrapOrThrow($result, $throwFirst);
    }

    protected static function unwrapOrThrow(ValidationResult $result, $throwFirst)
    {
        if ($result->isPassed()) {
            return (array)$result->validatedData();
        }

        $detail = $result->detail();
        $errorDetail = $throwFirst ? $detail[0] : $detail;

        throw new ValidateException(static::tipsByDetail($errorDetail));
    }

    protected static function tipsByDetail($detail)
    {
        return $detail;
    }
}
```

## 5. 这份样板里最关键的 4 个点

### 5.1 非 throw 方法保留旧数组 envelope

如果项目历史上大量代码已经依赖：

```php
$result['count']
$result['errors']
$result['detail']
$result['validated_data']
```

那就在适配层里统一：

```php
parent::validate(...)->toArray()
```

不要改 core。

### 5.2 throw 方法继续返回 payload 数组

如果项目历史上大量代码写的是：

```php
$payload = Validator::validateAndNormalizeOrThrow(...);
```

那就继续让适配层返回：

- `validatedData()`

而不是把调用方全改成操作 `ValidationResult`。

### 5.3 legacy 规则与 alias 留在项目内

比如：

- `egt`
- `elt`
- `lenMin`
- `lenMax`
- `trimmedRequiredString`
- `nullableIf`

都不应该要求 core 保留。

项目需要时：

- 纯改名 -> 放 `ruleAliases`
- 行为不完全等价 -> 自己写项目规则类，挂到 `extraRules`

### 5.4 中文错误提示与异常翻译留在项目内

不同项目：

- 语言不同
- 抛错模型不同
- 交互容忍度不同

所以：

- 中文文案
- `ValidateException`
- `tipsByDetail()`

都应当是项目适配层职责。

## 6. 跨字段比较规则的接法

core canonical 现在推荐：

```php
'start_at:开始时间' => 'time',
'end_at:结束时间' => 'timeAfterOrEqualField:start_at',
```

也就是说：

- 比较规则参数只传字段路径
- `$rule` 展示值由字段声明自动取“开始时间”

如果某个旧项目还在用：

```php
timeAfterOrEqualField:start_at,开始时间
```

则建议：

- 在项目适配层自行兼容
- 不要要求 core 继续把这种写法作为 canonical

---

## 7. 接入检查清单

一个新项目接入前，至少先明确这几件事：

1. 是否要保留旧数组 envelope
2. 是否要保留 `validateOrThrow()` / `validateAndNormalizeOrThrow()`
3. 是否有 legacy rule name 需要 alias
4. 是否有 legacy 行为需要单独 rule 类承接
5. 是否有 ORM / 业务规则需要放在项目层
6. 错误提示语言与异常模型是什么
7. 是否仍需兼容旧的跨字段比较参数格式

---

## 8. 推荐迁移顺序

推荐按下面顺序迁移：

1. 先引入 core 包
2. 建项目适配层 `Validator`
3. 先把旧 envelope 和 `*OrThrow` 接回
4. 再逐步把 legacy rule / alias / message 收口到适配层
5. 最后再逐步减少项目内已无必要的旧兼容逻辑

原则是：

> 先把项目接稳，再慢慢瘦兼容层；不要一上来要求所有旧调用同时改心智。
