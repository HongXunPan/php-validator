# 从其他验证库迁移

本文说明从 Laravel Validator、Symfony Validator、Respect Validation、Valitron、Rakit Validation 等验证库迁移到 `hongxunpan/validator` 时的写法改造和迁移成本。

本文不追求一比一兼容。迁移前应先判断：你是要替换整个验证体系，还是只把某些项目级规则沉淀为共享 core。

---

## 1. 迁移前判断

### 1.1 适合迁移的内容

适合迁移到本包 core 的能力：

- 基础字段存在性：`required`、`nullable`、`default`；
- 字符串归一化：`trim`、`blankToNull`；
- 基础类型和数值转换：`string`、`int`、`positiveInt`、`nonNegativeInt`；
- 基础长度、数量、枚举判断；
- 条件 required / nullable / prohibited；
- 当前字段与引用字段的数值或时间比较；
- 可复用的项目级业务规则。

### 1.2 不建议直接迁入 core 的内容

以下能力更适合留在项目 adapter 或框架层：

- 数据库 `unique / exists`；
- 上传文件、图片、MIME、尺寸；
- HTTP Request / Response；
- Laravel FormRequest；
- Symfony Entity / DTO Attribute Constraint；
- 多语言资源包和框架翻译系统；
- 依赖 ORM、缓存、外部服务的业务规则。

---

## 2. 迁移成本分级

| 成本 | 类型 | 说明 |
| --- | --- | --- |
| 低 | 基础字段规则 | `required`、`nullable`、`trim`、`minLength`、`maxLength`、`in` 等通常只需要改规则名或参数格式 |
| 中 | 条件规则 | `required_if`、`prohibited_if`、`nullable_if` 等需要改成更明确的规则名和严格参数 |
| 中 | 归一化规则 | 需要确认最终读取 `validatedData()`，而不是原始输入 |
| 中 | 自定义规则 | 从回调或框架 rule 改成 Rule 类，初期有成本，长期收益更高 |
| 高 | 数据库规则 | 不建议放进 core，通常要写 adapter rule 或留在框架层 |
| 高 | 文件上传 | 不建议直接迁入 core，适合项目 adapter |
| 高 | 对象 / DTO / Entity Constraint | 如果原系统围绕对象元数据建模，迁移心智差异较大 |

## 3. 从 Laravel Validator 迁移

### 3.1 基础规则

Laravel 写法：

```php
$validated = $request->validate(array(
    'name' => 'required|string|max:20',
));
```

本包推荐写法：

```php
$result = DemoValidator::validateAndNormalize(
    array('name' => ' Alice '),
    array('name:姓名' => 'required|trim|string|maxLength:20')
);

if ($result->isPassed()) {
    $data = $result->validatedData();
}
```

差异：

- 本包不会自动读取 Request；
- 本包不会自动抛 Laravel `ValidationException`；
- 本包输出 `ValidationResult`；
- 如果需要旧数组 envelope，可由 adapter 调用 `toArray()`；
- `max` 这类 Laravel 复合语义规则，迁移时应按类型拆成 `maxLength`、`maxItems` 或数值比较规则。

### 3.2 条件规则

Laravel / Laravel-like 常见写法：

```text
required_if:type,guest
```

本包不建议保留这种裸字符串逗号拆分心智。应迁移为更明确的条件规则，例如：

```text
requiredIfEq:type,"guest"
```

如果期望值是数字、布尔或 null，应使用严格字面量：

```text
requiredIfEq:status,1
requiredIfEq:enabled,true
requiredIfEq:deleted_at,null
```

集合判断可迁移为：

```text
requiredIfIn:type,["guest","member"]
```

收益是避免把 `"1"`、`1`、`true`、`"true"` 混在一起。

### 3.3 数据库规则

Laravel 写法：

```text
unique:users,email
exists:users,id
```

不建议把这类规则直接放进本包 core。

推荐做法：

1. core 只负责字段存在性、格式、基础归一化；
2. 项目 adapter 或业务层承接 ORM 查询；
3. 若多个项目都需要，再做独立 adapter 包，而不是让 core 依赖数据库。

### 3.4 FormRequest 与自动响应

Laravel 的 FormRequest 同时承接：

- 授权；
- 请求输入准备；
- 验证规则；
- 错误响应；
- controller 注入。

本包不替代这些能力。迁移时应保留项目层 adapter：

```php
$result = DemoValidator::validateAndNormalize($input, $rules);

if ($result->isFailed()) {
    // 在项目 adapter 中转换成框架需要的异常或响应
}
```

## 4. 从 Rakit Validation 迁移

Rakit 更接近 Laravel-like standalone 验证器，因此基础规则迁移成本较低，主要差异在：

- 本包使用 `ValidationResult`；
- 本包强调归一化后的 `validatedData()`；
- 条件规则参数更严格；
- 文件上传和 Laravel-like 复杂规则不进入 core；
- 自定义规则推荐迁移为结构化 Rule 类。

示例：

```php
$result = DemoValidator::validateAndNormalize(
    $_POST,
    array(
        'email:邮箱' => 'required|trim',
        'age:年龄' => 'nullable|positiveInt',
    )
);
```

如果原 Rakit 规则依赖 `ModifyValue`、`BeforeValidate` 等 hook，需要按本包规则阶段重新归类：

- 输入预处理：adapter；
- 缺失值创建：`AbstractMissingValueCreationRule`；
- present 值归一化：`AbstractPresentValueNormalizationRule`；
- present 值转换：`AbstractPresentValueTransformRule`；
- 断言：`AbstractPresentValueAssertionRule`。

## 5. 从 Valitron 迁移

Valitron 写法偏简单直接：

```php
$v = new Valitron\Validator(array('name' => 'Alice'));
$v->rule('required', 'name');
$v->rule('lengthMax', 'name', 20);
```

迁移到本包后，规则声明更集中在字段 map 中：

```php
$result = DemoValidator::validateAndNormalize(
    array('name' => 'Alice'),
    array('name:姓名' => 'required|maxLength:20')
);
```

迁移成本：

- 基础规则：低；
- 自定义回调：中，需要改成 Rule 类；
- 多上下文规则：中，可用不同 Validator 子类或不同规则 map 承接；
- 复杂错误消息：中，需要迁到 `ruleMessages` 或 adapter。

## 6. 从 Respect Validation 迁移

Respect 常见心智是 fluent 单值规则：

```php
v::numericVal()->positive()->between(1, 255)->isValid($input)
```

本包迁移后，心智变为字段级 payload 规则：

```php
$result = DemoValidator::validateAndNormalize(
    array('count' => '12'),
    array('count:数量' => 'positiveInt|gte:1|lte:255')
);
```

差异：

- Respect 更适合单值 fluent rule；
- 本包更适合数组输入、字段路径和输出数据；
- Respect 的大量规则不应机械迁入；
- 建议只迁移项目真实用到的高价值规则。

如果原 Respect 规则组合很多，建议先按业务场景整理成自定义 Rule，而不是逐条照搬 fluent 链。

## 7. 从 Symfony Validator 迁移

Symfony Validator 通常围绕：

- Constraint；
- ConstraintValidator；
- Attribute / YAML / XML / PHP metadata；
- Entity / DTO；
- ConstraintViolationList；

本包围绕：

- array input；
- 字段路径；
- 字符串 DSL；
- Rule 类；
- `ValidationResult`；
- `validatedData()`。

因此 Symfony 迁移不是简单改规则名，而是建模方式改变。

适合迁移的情况：

- 某些接口 payload 不适合上升到 Entity / DTO；
- 需要兼容旧 PHP 或非 Symfony 环境；
- 需要把输入归一化和校验统一放在轻量 core 中。

不适合迁移的情况：

- 现有系统已经稳定依赖 Attribute Constraint；
- 违规信息、分组、级联对象验证已经深度使用 Symfony 生态；
- 验证目标主要是对象图而不是数组 payload。

## 8. 自定义规则迁移模板

### 8.1 原始回调式规则

旧写法可能是：

```php
function isValidCode($value) {
    return preg_match('/^[A-Z0-9]+$/', $value);
}
```

迁移后推荐写成 Rule 类：

```php
use HongXunPan\Validator\Context\RuleContext;
use HongXunPan\Validator\Result\RuleResult;
use HongXunPan\Validator\Rule\AbstractPresentValueAssertionRule;

class CodeRule extends AbstractPresentValueAssertionRule
{
    const KEY = 'code';
    const MESSAGE = '$paramName 格式不正确';

    public static function validate(RuleContext $context)
    {
        if (!is_string($context->value())) {
            return RuleResult::fail($context->value());
        }

        if (!preg_match('/^[A-Z0-9]+$/', $context->value())) {
            return RuleResult::fail($context->value());
        }

        return RuleResult::pass($context->value());
    }
}
```

再在 Validator 子类中声明：

```php
class DemoValidator extends Validator
{
    protected static $extraRules = array(
        'code' => CodeRule::class,
    );
}
```

### 8.2 条件自定义规则

如果规则需要读取其他字段，优先继承条件或跨字段基类，而不是手写原始数组读取：

- `AbstractConditionalFieldPresenceRule`
- `AbstractConditionalPresentValueGuardRule`
- `AbstractReferencedFieldCompareRule`

这样可以复用字段引用解析、依赖字段读取、缺失跳过、显示名渲染等公共逻辑。

---

## 9. 推荐迁移顺序

1. 先迁移基础归一化规则：`trim`、`blankToNull`、`default`；
2. 再迁移基础断言：`required`、`string`、`int`、`minLength`、`maxLength`；
3. 再迁移枚举和列表规则；
4. 再迁移条件规则，并统一改成严格字面量参数；
5. 最后迁移自定义业务规则；
6. 数据库、文件上传、HTTP 响应保留在 adapter 层。

不要一开始追求完全替换旧验证器。更稳的方式是先选一个高价值、低耦合的 payload 场景做试点，再逐步扩大覆盖范围。
