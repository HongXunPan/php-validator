# 与同类验证库对比

本文用于帮助使用者判断 `hongxunpan/validator` 是否适合当前项目。

注意：同类项目的版本、依赖、PHP 要求与规则覆盖会持续变化；本文只描述选型维度与能力差异，不承诺第三方项目的实时版本信息。需要做正式依赖决策时，应以各项目官方 README、文档与 `composer.json` 为准。

---

## 1. 本包定位

`hongxunpan/validator` 的核心定位是：

- 框架无关；
- 低依赖；
- 面向数组输入和接口 payload；
- 同时承接校验、归一化、默认值、条件规则和跨字段依赖读取；
- 把自定义规则作为第一优先级，而不是只追求内置规则数量。

它更像一个可长期演进的验证内核，而不是某个框架验证器的无缝替代品。

---

## 2. 横向快速对比

| 项目 | 更适合的场景 | 相对本包的优势 | 本包的差异点 |
| --- | --- | --- | --- |
| Laravel Validator | Laravel 应用内的请求、表单、FormRequest、数据库唯一性、文件上传 | 官方集成、规则丰富、HTTP 错误响应成熟、生态最强 | 本包无 Laravel 依赖，更适合共享 core、旧 PHP、非 Laravel 项目和自定义规则治理 |
| Symfony Validator | Entity / DTO / Attribute / Constraint 驱动的对象验证 | 对象约束成熟、Violation 模型完整、Symfony 生态强 | 本包更偏数组输入、字段路径、归一化输出和接口 payload 生命周期 |
| Respect Validation | fluent 单值校验、大量独立规则组合 | 规则数量多、链式表达自然、社区成熟 | 本包更关注多字段 payload、输出数据、条件规则和跨字段 prepared value |
| Valitron | 简单轻量、低依赖、快速表单验证 | API 简单、成熟、学习成本低 | 本包结构更强，Rule / Context / Result 边界更适合长期维护自定义规则 |
| Rakit Validation | Laravel-like standalone 验证器 | Laravel-like 写法、数组和文件规则更接近表单验证 | 本包不追求 Laravel-like，而是追求低依赖内核、规则生命周期和 adapter 分层 |
| Particle Validator | 可读的 fluent array validator | 链式声明清晰、上下文能力较完整 | 本包保留字符串 DSL，同时把规则执行阶段和归一化输出作为核心设计 |

## 3. 能力维度对比

### 3.1 自定义规则扩展

本包把自定义规则作为第一优先级。

使用侧推荐通过：

- `RuleInterface`；
- `AbstractRule`；
- 各类规则基类，例如：
  - `AbstractPresentValueNormalizationRule`
  - `AbstractPresentValueGuardRule`
  - `AbstractPresentValueTransformRule`
  - `AbstractPresentValueAssertionRule`
  - `AbstractCrossFieldAssertionRule`
  - `AbstractConditionalFieldPresenceRule`
  - `AbstractConditionalPresentValueGuardRule`
  - `AbstractReferencedFieldCompareRule`
- `RuleContext`；
- `RuleResult`；
- `RuleArgumentParserInterface`；

来承接规则实现。

这意味着自定义规则不是一个孤立回调，而是能明确声明：

- 规则 key；
- 默认消息；
- 参数解析器；
- 当前字段值；
- 原始字段值；
- 依赖字段值；
- 是否中断后续规则；
- 是否改变输出值；
- 当前值是否存在。

相比之下：

- Laravel / Rakit 更适合直接复用已有规则生态；
- Respect 更适合 fluent 单值规则组合；
- Symfony 更适合 Constraint / Validator 对象模型；
- 本包更适合把项目级业务规则沉淀为稳定、可测试、可迁移的 Rule 类。

### 3.2 校验与归一化

本包不是只做断言。

内核中存在明确的数据生命周期：

1. 缺失值创建，例如 `default`；
2. present value normalization，例如 `trim`、`blankToNull`；
3. present value guard，例如 `nullable`、`nullableIf*`；
4. present value transform，例如 `positiveInt`、`nonNegativeInt`、`formatTime`；
5. present value assertion，例如 `minLength`、`maxLength`、`in`；
6. cross-field assertion，例如 `gtField`、`timeAfterOrEqualField`。

这使得 `validatedData()` 可以直接返回归一化后的数据，而不是只返回原始输入是否通过。

### 3.3 跨字段依赖读取

本包的一个关键差异是：跨字段比较规则优先读取依赖字段的 prepared value。

例如：

- `min_value` 先被 `trim|positiveInt` 归一化为整数；
- `current_value` 再执行 `gtField:min_value`；
- 比较规则读取的是 `min_value` 的归一化结果，而不是原始字符串。

这能减少级联脏错误，也更贴近接口入参最终可用值。

### 3.4 条件规则参数

本包倾向于结构化参数和严格字面量。

例如条件规则不鼓励长期使用 `field,value` 这种模糊参数，而是通过 argument parser 把字段引用、期望值、期望值集合解析成明确对象。

收益：

- 类型语义更清楚；
- 可以区分字符串、数字、布尔、null；
- 减少历史规则里逗号拆分、裸字符串、弱比较造成的歧义。

成本：

- 从 Laravel / Rakit 迁移条件规则时，写法改造会更明显；
- 使用者需要理解 rule argument parser 的边界。

### 3.5 依赖复杂度

本包当前运行时只要求 PHP，目标是保持 core 低依赖。

这不是说所有验证能力都应该塞进 core。相反，以下能力更适合放在 adapter 层：

- ORM `unique / exists`；
- 上传文件、MIME、图片尺寸；
- HTTP 请求对象；
- 框架错误响应；
- 业务数据库查询；
- 多语言资源包。

这种取舍降低了 core 的依赖复杂度，也意味着它不会像 Laravel / Symfony 那样开箱即用地覆盖完整应用场景。

---

## 4. 优点

1. **自定义规则优先**：规则类、上下文、结果对象、参数 parser 都围绕扩展而设计。
2. **低依赖**：适合共享包、旧项目和非框架项目。
3. **校验 + 归一化一体化**：最终输出可以直接来自 `validatedData()`。
4. **跨字段 prepared value**：依赖字段先完成自身处理，再被其他字段读取。
5. **公开契约边界清楚**：README 和稳定性文档明确区分公开面与 internal。
6. **adapter 友好**：core 不强行接管 HTTP、ORM、上传文件等项目层职责。

---

## 5. 缺点与不适用场景

1. **内置规则数量还少**：无法和 Laravel、Symfony、Respect 等成熟生态相比。
2. **pre-1.0 阶段**：公开契约已开始收口，但还不应按 1.x 稳定库预期使用。
3. **迁移不追求无缝**：特别是条件规则、DB 规则、文件规则、错误响应。
4. **现代 PHP 表达受限**：为了兼容旧 PHP，不能依赖 Attribute、enum、typed property 等现代语法作为 core 基线。
5. **对象验证不是主战场**：如果核心需求是 Entity / DTO Constraint，Symfony Validator 更合适。
6. **生态信任需要时间**：下载量、社区反馈、长期 issue 沉淀都还无法和成熟项目相比。

---

## 6. 选型建议

### 6.1 推荐选择本包

当你需要：

- 在多个项目之间共享验证规则；
- 支持旧 PHP 环境；
- 避免把 Laravel / Symfony 依赖带进 core；
- 把业务校验规则沉淀为结构化 Rule 类；
- 在同一条流水线中完成输入归一化、条件判断、跨字段读取和输出数据生成。

### 6.2 不推荐选择本包

当你主要需要：

- Laravel FormRequest；
- 自动 422 JSON 响应；
- `unique / exists` 等数据库规则；
- 上传文件和图片验证；
- Entity / DTO Attribute Constraint；
- 尽可能多的开箱规则。

这些场景更适合直接使用对应框架或成熟验证库，本包可以作为局部 core 或 adapter 背后的基础能力，而不是强行替换整个生态。
